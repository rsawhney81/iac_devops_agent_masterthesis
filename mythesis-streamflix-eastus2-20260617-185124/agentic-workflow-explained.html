Multi-Agent IaC Orchestration - How the System Works

A "1 + 9" architecture: one central IaC Workflow Orchestrator coordinating nine specialised agents through a governed, human-in-the-loop pipeline. The application to deploy is an input to the system, not a fixed target - StreamFlix is the demonstrator payload.

2. What each agent does

AgentResponsibility
IaC Workflow OrchestratorCentral coordinator. Seeds the global canon, runs preflight (auth, SKU, version, role checks), drives each phase in order, manages human-in-the-loop approvals, and enforces the terminal finalization gate.
Requirement NormalizerTurns the natural-language request + context.json into a validated specification, and emits a per-request requirement checklist (canon instantiated against this request). Three bounded passes: extraction, completion, feasibility.
Architecture MapperProduces the architecture model and the PlantUML source/diagram for the requested topology.
Module DiscovererSelects the Terraform modules or raw resources needed to realise the architecture.
Module ParameterizerMaps specification + context values onto module inputs; produces the parameterised configuration (including an application-deploying bootstrap).
Terraform AssemblerGenerates the deployable Terraform files and cloud-init that clones and serves the actual application - not merely a web server.
CI/CD GeneratorGenerates the GitHub Actions workflow (OIDC, keyless auth) for the pipeline.
Validator & GovernorRuns fmt/init/validate/plan/apply, enforces governance, and reports errors and fixes to the Knowledge Management Agent.
POC Documentary AgentAssembles the evidence package (this report's sibling) from measured artifacts; embeds the architecture PNG; reports only measured metrics, marking uninstrumented values null.
Knowledge Management AgentAppends durable lessons to the global canon so future runs start informed; maintains the repo-local error knowledge base.

4. Design principles

PrincipleHow it is realised
Application-agnosticThe app (repo, branch, bootstrap) comes from the request and context.json; nothing is hardcoded to a specific application.
Explicit specificationThe Requirement Normalizer emits a checkable per-request specification (unambiguous, feasible, traceable, machine-checkable) - distinct from unstructured prompting.
Cheap-zone validationFeasibility checks run before any resource exists, so failures cost nothing to recover.
Governed completionThe terminal finalization gate makes evidence, verification, and learning mandatory - "VM is up" is not a stopping condition.
Cross-run learningThe Knowledge Management Agent writes durable lessons to a global canon that travels across projects.
